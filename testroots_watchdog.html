Understanding how (much) Developers test.



The premise of TestRoots is to gain a deep understanding of how software is tested in the real world, and then suggest improvements based upon this thorough analysis. With WatchDog, we are interested in how you, as a Java developer, utilize Eclipse and Junit. Not what you use it for. Hence, the data WatchDog measures is strictly numerical: We never transfer any content (not file names, and not even project names). You can trust us on this, we are nice guys. Or read on, what WatchDog, an unobtrusive, open-sourced Eclipse plugin developed at TU Delft, really transfers.



I am a techie. Please, tell me what you transfer and how you transfer it. I'll make sense of it myself.

WatchDog centers around the concepts of intervals. Based on what event WatchDog receives from the Eclipse IDE, it closes or creates a new interval. We have intervals for when you start Eclipse, when you change focus to Eclipse, when you change your perspective, when you run Junit tests and when you read or type code. When you read or type code, we see if we can figure out whether that's production or test code. When you type, we try to figure out how much code you changed in each typing interval (by calculating the Levensthein distance). That's all we do. So we will never see one letter that you type.

Still not convinced? Alright, here's a JSON-ified TypingInterval (the most interesting one, really) as we transfer it to our server: 

{ "endingDocument" : { "sloc" : 3, "dt" : "un" }, "diff" : 425, "doc" : { "sloc" : 31, "dt" : "un" }, "it" : "ty", "ts" : NumberLong("1408110488025"), "te" : NumberLong("1408110508039"), "ss" : NumberLong("-7490789694621659010"), "uid" : "935d7573c056c424f668d935ef18b4a0bf96c365", "pid" : "efdcb880844893f847d70907174315d95c50caa6" }
